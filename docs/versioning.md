# Peak Log バージョン更新作業ガイド（Claude Code参照用）

このドキュメントは、Peak Log の新しいバージョンを追加するときに、Claude Code が参照するための共通ガイドです。  
新しいバージョンを指示するときは、この方針に従って作業してください。

---

## 目的

- アプリのバージョンを一貫して管理する
- Git にバージョン履歴が自然に残るようにする
- 各バージョンの変更内容を簡単なテキストで記録する
- Settings ページのバージョン表示に反映する
- 運用はできるだけシンプルに保つ

---

## 基本方針

### 1. バージョン番号の正本
- バージョン番号の正本は **`package.json` の `version`** とする
- バージョン番号を他ファイルで重複管理しない
- Settings ページではハードコードせず、`package.json` の version を参照する

### 2. 変更履歴
- 変更履歴はプロジェクトルートの **`CHANGELOG.md`** に記録する
- 1バージョンにつき、簡潔な箇条書きで記録する
- 細かい実装詳細ではなく、**ユーザーから見た変更内容**を優先してまとめる

### 3. Settings ページの表示
- Settings ページの既存のバージョン表示箇所に現在の version を表示する
- 表示元は `package.json` の `version`
- 必要なら `src/lib/app-version.ts` のような薄いラッパーを使ってよい

### 4. Git ベースで調査する
- 新しいバージョンを追加するときは、**必ず Git 履歴と差分を確認してから** CHANGELOG を書く
- 推測で変更内容を書かない
- コミットメッセージだけでなく、必要に応じてファイル差分も確認する

---

## バージョン更新時の標準作業手順

新しいバージョンを追加するときは、必ず以下の順序で進めること。

1. `package.json` の現在 version を確認する
2. `CHANGELOG.md` の既存内容を確認する
3. 今回の対象範囲となる基準コミットまたは前バージョン時点を特定する
4. 基準点から `HEAD` までのコミット履歴と差分を調査する
5. 変更内容を、ユーザー視点で意味のある粒度に要約する
6. `package.json` の `version` を新しい番号に更新する
7. `CHANGELOG.md` に新しいバージョンを追記する
8. Settings ページのバージョン表示が反映されることを確認する
9. 必要なら関連する import / utility を調整する
10. 最後にコミットする

---

## CHANGELOG 記述ルール

### 記述方針
- 箇条書きは簡潔でよい
- 実装の内部事情ではなく、プロダクト上の変更が分かるようにする
- 同じ意味の変更を細かく分割しすぎず、適度にまとめる
- 以前のバージョンと重複しないようにする

### まとめ方の観点
以下の観点で整理すると分かりやすい。

- 機能追加
- UI改善
- 操作性改善
- 表示改善
- 導線改善
- History / Home / Activities / Settings など画面別の改善
- 細かな不具合修正や挙動調整

### 避けること
- Git のコミットメッセージをそのまま羅列すること
- 技術詳細だけで終わること
- 実際に確認していない内容を書くこと

---

## CHANGELOG の基本フォーマット

```md
# Changelog

## x.y.z - YYYY-MM-DD

- 変更内容1
- 変更内容2
- 変更内容3

## a.b.c - YYYY-MM-DD

- 変更内容1
- 変更内容2
````

---

## Settings ページの扱い

* 既存のバージョン表示箇所を利用する
* 表示文言は既存UIトーンに合わせる
* version はハードコードしない
* `package.json` の version を参照する実装を維持する

---

## 実装時の注意

* バージョン管理の正本は `package.json` のみ
* `CHANGELOG.md` はシンプルな手書き運用を前提とする
* 将来は `package.json` と `CHANGELOG.md` を更新すれば済む状態を保つ
* 既存の UI トーンを崩さない
* 不要な仕組みや自動化を追加しない
* バージョン更新のたびに、必ず Git 調査を先に行う

---

## 作業完了時に必要な報告

作業完了後は、以下を簡潔に報告すること。

1. 今回のバージョンに含めた変更内容の要約
2. 更新したファイル一覧
3. Settings ページで version をどこから参照しているか
4. 作成したコミットメッセージ

---

## コミット方針

作業完了後は、変更内容が分かるコミットメッセージでコミットすること。
例:

* `chore: bump version to x.y.z and update changelog`
* `feat: release x.y.z`
* `chore: update app version and changelog`

---

## 新しいバージョン作業を依頼するときのテンプレート

以下を埋めて使うこと。

```md
Peak Log のバージョンを **{{NEW_VERSION}}** に更新してください。

この作業では、必ず `Peak Log バージョン更新作業ガイド` を参照して進めてください。

## 今回の前提
- 前バージョン: **{{PREVIOUS_VERSION}}**
- 今回の対象範囲: **{{RANGE_DESCRIPTION}}**
- 基準コミット: **{{BASE_COMMIT_OR_TAG}}**
- 完成日: **{{RELEASE_DATE}}**

## やってほしいこと
1. Git 履歴と差分を確認して、前バージョンから現在までの変更内容を調査
2. 変更内容をユーザー視点で要約
3. `package.json` の `version` を **{{NEW_VERSION}}** に更新
4. `CHANGELOG.md` に **{{NEW_VERSION}}** を追記
5. Settings ページのバージョン表示に反映されることを確認
6. 必要なら表示や参照コードを調整
7. 最後にコミット

## CHANGELOG の記載方針
- 実際に調査した内容に基づいて書く
- 技術詳細ではなく、プロダクト上の変更が分かる形で簡潔にまとめる
- 前バージョンと重複しないようにする

## 作業完了後に報告してほしいこと
1. {{NEW_VERSION}} に含めた変更内容の要約
2. 更新したファイル一覧
3. Settings ページでどのように version を参照しているか
4. 作成したコミットメッセージ
```

---

## 補足

過去バージョンの整理や、どのコミット範囲をどのバージョンに含めるかは、
必ず今回の依頼文に明記された内容を優先してください。

不明な場合は、まず既存の `CHANGELOG.md` と Git 履歴から整合する範囲を確認し、
推測ではなく事実ベースでまとめてください。
